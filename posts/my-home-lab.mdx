A homelab isn't just servers in a closet. It's a playground, a quiet production system for the stuff I actually use, and a way to keep my own data on my own machines. This is a tour of what I'm running.

For the physical machines, see [Homelab Hardware](/blog/homelab-hardware).

## The hardware

My cluster runs on compact, power-efficient mini PCs:

| Node | Role | CPU | RAM | Purpose |
|------|------|-----|-----|---------|
| intel-c-firewall | Control Plane | 4 cores | 8 GB | Kubernetes API, etcd (no workloads) |
| amd-w-minipc | Worker | 16 cores | 13 GB | Primary workloads |
| intel-w-acemagic | Worker | 4 cores | 16 GB | Storage-heavy workloads |

The `intel-c-firewall` node runs as a VM on Proxmox, hosted on a Glovary mini PC that also runs an OPNsense VM (firewall) and Pi-hole DNS. The other two nodes (`amd-w-minipc` and `intel-w-acemagic`) are bare metal mini PCs.

**Total Resources:** 24 CPU cores, ~35 GB RAM, ~2.1 TB storage

All three sit on a single unmanaged switch. No VLANs. No segmentation. Flat and boring, which is what I wanted.

The whole cluster idles at around 50W. That's why I can leave it on. The electricity bill barely notices.

## The OS: Talos Linux

I'm not running a normal Linux distro. I'm running [Talos Linux](https://www.talos.dev/), which is built only for Kubernetes. There's no SSH. No shell. You can't log in. You manage it through an API.

That sounds annoying until you live with it. Then it's calm. Nothing to forget to patch, nothing to half-configure. You write YAML, you run `talosctl`, the node does what you said. Upgrades roll through one node at a time. Nothing goes down.

Current versions:
- Talos: v1.11.5
- Kubernetes: v1.34.1
- containerd: v2.1.5

## GitOps with Flux

The cluster is run from a Git repo. [Flux v2](https://fluxcd.io/) watches the repo and applies whatever's there. If I want to change something, I commit. Secrets are encrypted with [SOPS](https://github.com/getsops/sops) and [age](https://github.com/FiloSottile/age), with a few sitting in [1Password](https://1password.com/).

<GitHubRepo repo="eduuh/kube-homelab" description="GitOps repository for my Kubernetes homelab - Talos Linux, Flux, and all the YAML" />

```
cluster/
├── apps/           # Application deployments
├── networking/     # Ingress, DNS, load balancing
├── storage/        # Longhorn configuration
├── cert-manager/   # TLS certificates
├── monitoring/     # Prometheus stack
└── flux-system/    # Flux controllers
```

Push, walk away. Flux does the rest.

## Networking

### Load balancing: MetalLB

There's no cloud load balancer here, so I use [MetalLB](https://metallb.universe.tf/) in Layer 2 mode to hand out IPs to services:

- IP Pool: `10.0.0.53-10.0.0.60`
- Ingress Controller: `10.0.0.54`
- Pi-hole DNS: `10.0.0.55`

### Ingress: nginx

HTTP traffic comes through [ingress-nginx](https://kubernetes.github.io/ingress-nginx/) as a DaemonSet. Every service gets HTTPS via Let's Encrypt, a subdomain on `*.eduuh.com`, and timeouts that won't choke on a big upload.

### DNS: Pi-hole + CoreDNS

DNS is split in two:

```
Client → Pi-hole (10.0.0.55) → CoreDNS (10.0.0.53) for *.eduuh.com
                             → Cloudflare (1.1.1.1) for public domains
```

Pi-hole blocks ads at the DNS level. CoreDNS handles internal names.

## Storage: Longhorn

My rule for the homelab is simple: I care about getting data back, not about staying up. Services can fall over. Data can't disappear. [Longhorn](https://longhorn.io/) fits that. It spreads block storage across nodes, keeps two or three copies of each volume, and takes snapshots on a schedule. Both ReadWriteOnce and ReadWriteMany work.

If a node dies, the data is still somewhere. Off-cluster backups are next on my list. I haven't sat down with Longhorn's backup features yet.

## What I self-host

### Media

The thing every homelab ends up running. [Jellyfin](https://jellyfin.org/) streams everything (the open-source Plex alternative). [qBittorrent](https://www.qbittorrent.org/) does the actual downloading via a web UI. [Sonarr](https://sonarr.tv/) automates TV, [Radarr](https://radarr.video/) automates movies, and [Prowlarr](https://prowlarr.com/) manages the indexers both of them talk to.

All five share one 200GB volume:

```
/media/
├── downloads/
├── tv/
└── movies/
```

Sharing a filesystem matters here. When Sonarr moves a finished download into `tv/`, it's a hardlink. Instant, no extra space.

### Productivity

[Immich](https://immich.app/) is my Google Photos replacement, currently around 100GB. [Nextcloud](https://nextcloud.com/) does file sync. [Vaultwarden](https://github.com/dani-garcia/vaultwarden) serves Bitwarden clients without sending my passwords anywhere. [n8n](https://n8n.io/) runs workflow automation. [Memos](https://usememos.com/) is a small note-taking app I use for quick captures.

### Utilities

[Glance](https://github.com/glanceapp/glance) is the dashboard with service monitoring (more on it below). [IT-Tools](https://github.com/CorentinTh/it-tools) is a grab bag of developer utilities I'd otherwise have to google. [Uptime Kuma](https://github.com/louislam/uptime-kuma) is the thing that pages me when something falls over.

### The dashboard

Glance pulls it all together:

![Glance dashboard showing all homelab services](/images/Screenshot%202026-01-14%20at%2019.18.11.png)

It shows whether things are alive, how fast they're responding, and groups them into Applications, Media, Monitoring, and Infrastructure. When something feels off, this is the first place I look.

## Watching the cluster

You can't fix what you can't see.

### Prometheus and Grafana

The [kube-prometheus-stack](https://github.com/prometheus-community/helm-charts/tree/main/charts/kube-prometheus-stack) bundles Prometheus, Grafana, AlertManager, and a stack of Kubernetes dashboards. Metrics stick around for 30 days.

### Extra exporters

An OPNsense exporter pulls firewall stats (CPU, memory, traffic per interface, WAN latency). A speedtest tracker runs every hour so I have a record of what my ISP is actually delivering. I built a few small Grafana dashboards on top: bandwidth use, up/down speeds, traffic per interface.

## What I'd tell past-me

Start simple. My first attempt had Ansible, Terraform, and a pile of VMs. Talos plus Flux replaced all of it and is calmer.

GitOps pays for itself. The cluster could die tonight and I could rebuild it tomorrow from the repo. When something breaks, `git log` says who did what.

Shared storage isn't optional for the media stack. Plan it before you start.

Add monitoring on day one. Bolting it on later is miserable.

Recovery beats uptime. It's me and my wife. If Jellyfin is down for a few hours, nothing burns down. What matters is that the data comes back. Once you accept that, a lot of decisions get easier.

## Hardware I want to add

What's running works. There's still room to grow.

A small NAS would handle bulk storage and live offsite-ish for backups. I have two 4TB 2.5" SSDs sitting around that could go into the cluster, plus a 1TB drive earmarked for OS work. There's an old Dell laptop with an 8th-gen i7 that would be perfect for Jellyfin transcoding. If I roll that in, the ACEMAGIC gets a new job somewhere else.

![Dell laptop internals showing the motherboard, 42Wh battery, and upgrade potential](/images/20260114_192856.jpg)

Splitting bulk media off from app data would help both performance and backups. More space also opens the door to self-hosting camera feeds for the smart home.

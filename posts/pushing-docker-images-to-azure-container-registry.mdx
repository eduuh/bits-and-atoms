I needed four Docker images from a monorepo, pushed to Azure Container Registry on every commit to main. Sounded simple. It wasn't.

## What I wanted

Four services, four images, all going to `wira360.azurecr.io` on autopilot. Tag each one with the git SHA so I can trace it back, plus `latest` for convenience. And no stored secrets — OIDC was supposed to handle auth.

## The workflow that actually shipped

```yaml
name: Build and Push to ACR

on:
  push:
    branches: [main]
  workflow_dispatch:

jobs:
  build-push:
    name: ${{ matrix.service }}
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        include:
          - service: sokoni-api
            dockerfile: products/sokoni/api/Dockerfile
          - service: sokoni-web
            dockerfile: products/sokoni/web/Dockerfile
          - service: payments-api
            dockerfile: services/payments/api/Dockerfile
          - service: admin-web
            dockerfile: platform/admin/web/Dockerfile

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to ACR
        uses: docker/login-action@v3
        with:
          registry: wira360.azurecr.io
          username: \${{ secrets.ACR_USERNAME }}
          password: \${{ secrets.ACR_PASSWORD }}

      - name: Build and push
        uses: docker/build-push-action@v6
        with:
          context: .
          file: \${{ matrix.dockerfile }}
          push: true
          tags: |
            wira360.azurecr.io/\${{ matrix.service }}:\${{ github.sha }}
            wira360.azurecr.io/\${{ matrix.service }}:latest
          cache-from: type=registry,ref=wira360.azurecr.io/\${{ matrix.service }}:cache
          cache-to: type=registry,ref=wira360.azurecr.io/\${{ matrix.service }}:cache,mode=max
```

`fail-fast: false` keeps a slow build from cancelling the others. The registry cache means repeat runs only ship the layers that changed.

## One-time Azure setup

You need an ACR first. Mine was already at `wira360.azurecr.io`. Turn on admin credentials:

```bash
az acr update --name wira360 --admin-enabled true
az acr credential show --name wira360
```

That gives you a username and password. Save them as GitHub secrets named `ACR_USERNAME` and `ACR_PASSWORD`.

## The OIDC rabbit hole

I really wanted keyless auth. No stored secrets, just OIDC. The setup looks clean. App registration, federated credential pointed at the repo, `AcrPush` role assigned, `azure/login@v2` in the workflow.

```bash
az ad app create --display-name "github-tracker-acr"
az ad sp create --id <appId>

az ad app federated-credential create \
  --id <appId> \
  --parameters '{
    "name": "github-tracker-main",
    "issuer": "https://token.actions.githubusercontent.com",
    "subject": "repo:eduuh/wira360:ref:refs/heads/main",
    "audiences": ["api://AzureADTokenExchange"]
  }'

ACR_ID=$(az acr show --name wira360 --query id -o tsv)
az role assignment create --assignee <appId> --role AcrPush --scope $ACR_ID
```

<Callout type="warning" title="Audience gotcha">
  The audience must be `api://AzureADTokenExchange`. If you use `api://AzureADTokenEndpoint` (which looks plausible), OIDC silently fails with `AADSTS700212`.
</Callout>

OIDC login itself worked. The Docker push kept failing with `insufficient_scope: authorization failed`.

Here's what's going on. After `azure/login@v2` signs you in, the next step is getting Docker credentials for the registry. The obvious move is `az acr login --name wira360`, which writes credentials to the default Docker context. Buildx runs in its own container driver, though, and doesn't see them.

So you switch to `az acr login --expose-token`, grab a short-lived access token, and hand it to `docker login`. Closer, but still no. Buildx's OAuth challenge with ACR expects either basic auth or a refresh token. A bare access token doesn't pass the challenge, and ACR comes back with `insufficient_scope`.

There are ways to make OIDC work with buildx (pass the token through buildx secrets, swap drivers) but they trade a lot of complexity for very little. Admin credentials in GitHub secrets are boring and they work. Boring won.

## Checking it worked

```bash
az acr repository list --name wira360
az acr repository show-tags --name wira360 --repository sokoni-api
```

Your git SHAs should be sitting there as tags.

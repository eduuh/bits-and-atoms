I've been setting up CI/CD for a monorepo and ran into enough surprises that writing them down felt useful. None of this is beginner stuff. Assume you've already written a workflow or two.

## Matrix builds with `include`

If you've got multiple services to build, the obvious move is one job per service. The better move is a matrix:

```yaml
strategy:
  fail-fast: false
  matrix:
    include:
      - service: sokoni-api
        dockerfile: products/sokoni/api/Dockerfile
      - service: payments-api
        dockerfile: services/payments/api/Dockerfile
```

Each entry in `include` becomes a set of variables you can use as `${{ matrix.service }}`, `${{ matrix.dockerfile }}`, and so on. Four entries, four parallel jobs, one job definition.

`fail-fast: false` is the small detail that matters. The default cancels every other job the moment one fails. For builds, you almost always want to see all the failures at once, not just the first.

## OIDC: when it works, and when it doesn't

OIDC lets Actions authenticate to cloud providers without long-lived secrets. AWS and GCP make it easy. Azure mostly works, but it has a sharp edge.

The setup is the usual dance: app registration, federated credential pointed at your repo, the right role assigned. The audience on the federated credential has to be `api://AzureADTokenExchange`. Not `api://AzureADTokenEndpoint`, which looks like it could be right but isn't. Use the wrong one and Azure returns `AADSTS700212` without a hint about what's wrong.

```yaml
permissions:
  id-token: write   # required for OIDC token exchange
  contents: read
```

You need that `permissions` block at the workflow or job level. Without `id-token: write`, the token request just fails silently.

## The buildx credentials problem

If you're using `docker/build-push-action`, you're using buildx. And buildx runs in its own container driver, which means it doesn't see Docker credentials from the runner's default context.

So this:

```yaml
# This does NOT work with buildx
- run: az acr login --name myregistry
```

And even this fails:

```yaml
# Also doesn't work — ACR's OAuth flow rejects bare access tokens
- run: |
    TOKEN=$(az acr login --name myregistry --expose-token --output tsv --query accessToken)
    docker login myregistry.azurecr.io --username 00000000-0000-0000-0000-000000000000 --password "$TOKEN"
```

What works is `docker/login-action@v3` with credentials that survive the OAuth challenge. That means basic auth (username and password) or a proper refresh token. For ACR, the simplest path is enabling admin credentials:

```yaml
- uses: docker/login-action@v3
  with:
    registry: myregistry.azurecr.io
    username: ${{ secrets.ACR_USERNAME }}
    password: ${{ secrets.ACR_PASSWORD }}
```

## Registry cache for faster builds

Without caching, every run starts from zero. With registry cache, unchanged layers come from the registry:

```yaml
- uses: docker/build-push-action@v6
  with:
    cache-from: type=registry,ref=myregistry.azurecr.io/myservice:cache
    cache-to: type=registry,ref=myregistry.azurecr.io/myservice:cache,mode=max
```

`mode=max` caches every layer, even the intermediate ones. The first run is the same speed as before. Every run after is faster, and how much faster depends on how much of your Dockerfile actually changes.

The `:cache` tag is just a name. The first push creates it.

## Tag with the SHA, not the branch

```yaml
tags: |
  myregistry.azurecr.io/myservice:${{ github.sha }}
  myregistry.azurecr.io/myservice:latest
```

If you only tag with branch names like `main` or `latest`, you can't roll back to a specific build. Tagging with the SHA gives you a direct line from what's running to the commit that produced it. You can always walk back from the registry to the source.

## `workflow_dispatch` for manual runs

```yaml
on:
  push:
    branches: [main]
  workflow_dispatch:
```

`workflow_dispatch` adds a "Run workflow" button in the Actions tab. Handy when you've fixed something on the infra side (like a broken ACR password) and want to re-run without pushing a junk commit.

## Checking what broke

```bash
# List recent runs for a specific workflow
gh run list --repo org/repo --workflow build.yml --limit 5

# See failed step logs
gh run view <run-id> --log-failed

# Re-run a failed workflow
gh workflow run build.yml --repo org/repo --ref main
```

When you're poking at a broken workflow, `gh` is much faster than the UI.

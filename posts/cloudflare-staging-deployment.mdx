Shipping to production is the easy part. The harder part is having a staging site that looks like production but isn't open to the world. Here's how my blog ended up with Cloudflare Pages on the front and Cloudflare Access guarding staging.

## What I wanted

Production at `eduuh.com`, public, indexed.

Staging at `staging.eduuh.com`, private, just for me before things go live.

Pages handles the deploys fine, but it makes everything public by default.<Footnote id={1}>Cloudflare Pages automatically creates preview deployments for every branch and pull request, which are publicly accessible by default.</Footnote> I wanted to lock staging behind a login without baking auth into the app.

## The pieces

Cloudflare Pages for hosting and deploys. Cloudflare Access for the login wall. GitHub Actions to push the builds. Email OTP because I didn't want to manage passwords.

## The deployment flow

```d2
direction: right

GitHub: {
  shape: rectangle
  main: main branch
  staging: staging branch
}

Actions: GitHub Actions {
  shape: rectangle
  build: Build
  deploy: Deploy
}

Cloudflare: {
  shape: rectangle
  pages: Cloudflare Pages
  access: Cloudflare Access
}

Sites: {
  shape: rectangle
  prod: eduuh.com
  stage: staging.eduuh.com
}

GitHub.main -> Actions.build: push
GitHub.staging -> Actions.build: push
Actions.build -> Actions.deploy
Actions.deploy -> Cloudflare.pages

Cloudflare.pages -> Sites.prod: main branch
Cloudflare.pages -> Sites.stage: staging branch

User: User {
  shape: person
}

User -> Sites.prod: Public
User -> Cloudflare.access: Protected
Cloudflare.access -> Sites.stage: Email OTP
```

The trick: Access catches requests to `staging.eduuh.com` before they ever hit the site. The app doesn't know auth exists. It all happens at the edge.

## The GitHub Actions workflow

The workflow ships from whichever branch you pushed to:

```yaml
name: Deploy to Cloudflare Pages

on:
  push:
    branches:
      - main
      - staging
  workflow_dispatch:
    inputs:
      environment:
        description: 'Environment to deploy to'
        required: true
        default: 'staging'
        type: choice
        options:
          - staging
          - production

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci
      - run: npm run build

      - name: Set environment
        id: env
        run: |
          if [[ "${{ github.ref }}" == "refs/heads/main" ]]; then
            echo "branch=main" >> $GITHUB_OUTPUT
          else
            echo "branch=staging" >> $GITHUB_OUTPUT
          fi

      - name: Deploy to Cloudflare Pages
        uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          accountId: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
          command: pages deploy out --project-name=${{ vars.CLOUDFLARE_PROJECT_NAME }} --branch=${{ steps.env.outputs.branch }}
```

Cloudflare Pages handles the routing: `main` goes to production, `staging` goes to staging.

## Setting up Cloudflare Access

Access sits in front of staging and asks for a login before letting anyone in. The site itself doesn't change.

In the Zero Trust dashboard, go to Access → Applications and create a Self-hosted app pointing at `staging.eduuh.com`.

For login, I use One-time PIN — email OTP. Under Settings → Authentication, add it as a login method. There's no OAuth to set up.

Then add a policy. The policy decides who's allowed in:

```json
{
  "name": "Allow email login",
  "decision": "allow",
  "include": [
    {
      "email_domain": {
        "domain": "gmail.com"
      }
    }
  ]
}
```

That lets anyone with a Gmail address request access. You can lock it down to a single email instead:

```json
{
  "include": [
    {
      "email": {
        "email": "myemail@gmail.com"
      }
    }
  ]
}
```

## Doing it from the API

The dashboard is fine but I'd rather script it. Same setup, over the API:

```bash
# Set your credentials
CF_TOKEN="your-api-token"
CF_ACCOUNT="your-account-id"
APP_ID="your-app-id"

# Create access policy
curl -X POST "https://api.cloudflare.com/client/v4/accounts/$CF_ACCOUNT/access/apps/$APP_ID/policies" \
  -H "Authorization: Bearer $CF_TOKEN" \
  -H "Content-Type: application/json" \
  --data '{
    "name": "Allow email login",
    "decision": "allow",
    "include": [{"email_domain": {"domain": "gmail.com"}}],
    "precedence": 1
  }'

# Link OTP identity provider
curl -X PUT "https://api.cloudflare.com/client/v4/accounts/$CF_ACCOUNT/access/apps/$APP_ID" \
  -H "Authorization: Bearer $CF_TOKEN" \
  -H "Content-Type: application/json" \
  --data '{
    "name": "Blog Staging",
    "domain": "staging.eduuh.com",
    "type": "self_hosted",
    "allowed_idps": ["your-otp-provider-id"],
    "auto_redirect_to_identity": true
  }'
```

## DNS

By default, custom domains on Pages point at the production branch. To make `staging.eduuh.com` serve the staging branch, you point its CNAME at the branch alias:

```bash
# Update DNS to point to staging branch alias
curl -X PATCH "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/dns_records/$RECORD_ID" \
  -H "Authorization: Bearer $CF_TOKEN" \
  -H "Content-Type: application/json" \
  --data '{"content":"staging.blog-2026-290.pages.dev"}'
```

The branch alias (`staging.your-project.pages.dev`) always serves the latest deploy on that branch.

## Locking down preview URLs

Pages also gives every deploy its own URL — things like `9fad24de.blog-2026-290.pages.dev`. To cover all of them, set up a wildcard Access app:

```bash
# Create wildcard Access app
curl -X POST "https://api.cloudflare.com/client/v4/accounts/$CF_ACCOUNT/access/apps" \
  -H "Authorization: Bearer $CF_TOKEN" \
  -H "Content-Type: application/json" \
  --data '{
    "name": "Blog All Previews",
    "domain": "*.blog-2026-290.pages.dev",
    "type": "self_hosted",
    "session_duration": "24h",
    "allowed_idps": ["your-otp-provider-id"],
    "auto_redirect_to_identity": true
  }'
```

That covers the branch alias (`staging.blog-2026-290.pages.dev`), one-off deploy URLs like `9fad24de.blog-2026-290.pages.dev`, and anything Pages spins up later.

## Token permissions

For the deployment workflow, you need a token with Cloudflare Pages → Edit.

For Access and DNS, the token needs more: Access: Apps and Policies → Edit, Access: Organizations, Identity Providers, and Groups → Edit, and Zone → DNS → Edit. The "Edit Cloudflare Zero Trust" template covers most of it; add DNS to it.

## Things that bit me

An empty policy list silently blocks everyone. The login page loads, you enter your email, nothing happens. Always have at least one allow policy.

The identity provider has to be linked to the app. Setting up OTP globally isn't enough — you also have to add it to `allowed_idps` on the application.

Tokens are scoped tight. A Pages deploy token won't touch Access. Make a separate token for each thing.

Custom domains default to the production branch. Adding the domain doesn't pin it to staging — that's a separate CNAME update to the branch alias.

## Cost

Free, all the way down. Pages free tier covers most blogs. Access is free for up to 50 users. GitHub Actions is free for public repos.

## What's next

A few things on my list. Service tokens so CI can hit staging without email. Bypass rules for health checks. Eventually, Terraform to keep Access config in source control.

## In the end

Everything that isn't production is locked down:

| URL | Branch | Protected |
|-----|--------|-----------|
| `eduuh.com` | main | Public |
| `staging.eduuh.com` | staging | Access (email OTP) |
| `*.blog-2026-290.pages.dev` | all previews | Access (wildcard) |

Pages plus Access gives me a real deployment pipeline with nothing to maintain. That's a hard combo to argue with.

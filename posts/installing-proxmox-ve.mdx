The [Glovary mini PC](/blog/homelab-hardware) in my cluster does double duty: it runs OPNsense as my firewall and hosts a VM that acts as the Kubernetes control plane. Running two unrelated workloads on one box only works because of a hypervisor — and for that, I reach for Proxmox VE.

Proxmox VE is a Debian-based open-source hypervisor with a clean web UI, KVM for full VMs, and LXC for lightweight containers. It's what I use when I want "a VM on that box" without the complexity of a full virtualization platform.

Here's how I install it.

## Prerequisites

<Callout type="info">
Proxmox needs a CPU with hardware virtualization support (Intel VT-x or AMD-V). Enable it in the BIOS before installing — otherwise the installer will run, but you won't be able to start VMs.
</Callout>

You'll need:

- A machine with at least 2 GB RAM (8 GB+ if you're running actual workloads)
- A USB stick (4 GB or larger) for the installer
- A wired network connection during install
- Five to ten minutes

## Step 1: download the ISO

Grab the latest ISO from the [Proxmox downloads page](https://www.proxmox.com/en/downloads/proxmox-virtual-environment/iso). I went with the current stable release.

Verify the SHA256 checksum before flashing — it takes thirty seconds and rules out a corrupted download:

<Terminal lines={[
  "$ shasum -a 256 proxmox-ve_8.2-1.iso",
  "a1b2c3d4...  proxmox-ve_8.2-1.iso"
]}/>

Compare against the checksum listed on the download page.

## Step 2: write the ISO to USB

On macOS I use `dd` directly. First find the USB device:

```bash
diskutil list
```

Look for your USB stick (mine showed as `/dev/disk4`). Unmount it, then write the ISO:

```bash
diskutil unmountDisk /dev/disk4
sudo dd if=proxmox-ve_8.2-1.iso of=/dev/rdisk4 bs=1M status=progress
```

<Callout type="warning">
Double-check the device path. `dd` will happily overwrite your main drive if you point it at the wrong disk. The `r` prefix (`rdisk4`) is macOS-specific and makes the write much faster.
</Callout>

A note on the block size: I use `bs=1M` (uppercase) because it works on both BSD `dd` (macOS) and GNU `dd` (Linux). The lowercase `bs=1m` works on macOS but errors out on Linux — see [dd Block Size: BSD vs GNU](/blog/dd-block-size-bsd-vs-gnu) for the full story.

On Linux the same command works against `/dev/sdX` (no `r` prefix needed). On Windows, [Rufus](https://rufus.ie/) in DD mode is the easiest option.

## Step 3: boot the installer

Plug the USB into the target machine and boot from it. On the Glovary I had to press `Del` to enter the BIOS, enable virtualization, and set the USB as the first boot device.

The Proxmox installer boots into a graphical menu. Pick **Install Proxmox VE (Graphical)**.

Walk through the prompts:

1. **EULA:** read it, scroll to the bottom, click **I agree**. You can't proceed without accepting.
2. **Target disk:** I use the internal NVMe. The default ext4 filesystem is fine for a single-disk setup. For anything with redundancy, pick ZFS and add mirrored disks.
3. **Country, timezone, keyboard:** set these correctly now; changing them later is a pain.
4. **Root password and admin email:** use a real email — Proxmox sends alerts here.
5. **Network config:** see below. This is the step that deserves the most attention on a multi-NIC firewall box.
6. **Hostname (FQDN):** I went with `proxmox.home.arpa`.

### Picking the management NIC on a 4-port firewall box

The Glovary has four Intel i226-V 2.5GbE ports. During install, Proxmox asks you to pick **one** NIC to bind the management interface to — this is the port you'll use to reach the web UI, and the one the host itself uses for IP connectivity.

The installer lists them as `eth0` through `eth3` (or in newer versions, predictable names like `enp1s0`, `enp2s0`, etc.). Before picking, I mapped out what every port is going to do once OPNsense is running. No VLAN tagging for now — each physical port is its own network, and segregation is enforced at the wire:

| Port | Role | Network | Subnet |
|------|------|---------|--------|
| `eth0` | **WAN**, uplink to ISP | public | from ISP |
| `eth1` | **Office LAN**, wired desks + work WiFi AP | trusted | `10.10.2.0/24` |
| `eth2` | **IoT & Guest**, untrusted devices + guest AP | isolated | `10.10.3.0/24` |
| `eth3` | **LAN**, internal homelab + Proxmox mgmt | infra | `10.10.1.0/24` |

A few things worth calling out about this layout:

- **WiFi for work** piggybacks on the Office LAN port. One AP plugged into `eth1` broadcasting a single work SSID on the `10.10.2.0/24` subnet. No VLAN magic needed.
- **IoT and Guest share a port** (combined on `10.10.3.0/24`). A second cheap AP on `eth2` with AP isolation enabled covers both the guest SSID and smart-home devices. They can't reach each other or the LAN. Firewall rules in OPNsense enforce that.
- **Only three internal networks fit** on a 4-port box when you want physical segregation. If I ever want Office and WiFi on separate segments, or Guest truly isolated from IoT, I'll need to either add a managed switch with VLANs or get a bigger firewall appliance. Today, three is plenty.

On my Glovary, I picked **eth3** (the rightmost port) as Proxmox's management interface. The other three ports stay untouched so I can pass them straight through to the OPNsense VM.

A few notes on the choice itself:

- **Pick a port that's physically cabled and reachable.** The installer tests the link before letting you continue. If the cable isn't plugged in, the option won't appear.
- **Label them all.** Four identical 2.5GbE ports look the same. I labelled each one (WAN / OFFICE / IOT / LAN) before starting the install.
- **Don't pick the WAN port for management.** Once Proxmox owns that interface, passing it through to OPNsense gets messy. Keep WAN and mgmt on separate NICs.

With eth3 as management, I set:

- **IP:** `10.10.1.10/24`
- **Gateway:** `10.10.1.1` (OPNsense, once it's running)
- **DNS:** `10.10.1.20` (Pi-hole, to be deployed) with `1.1.1.1` as a fallback

The actual OPNsense configuration (assigning each port as its own interface, firewall rules between segments, DHCP per network) lives in the next post. Proxmox only needs to know about the LAN (`10.10.1.0/24`) since that's where the host itself lives.

### Static vs dynamic on `10.10.1.0/24` (the LAN segment)

Each of the three internal networks needs its own internal layout. Office (`10.10.2.0/24`) and IoT/Guest (`10.10.3.0/24`) are mostly dynamic clients — I'll cover those in the OPNsense post. The LAN segment is the interesting one because it holds all the infrastructure.

The mistake I've made before is letting DHCP and static assignments overlap — DHCP hands out an address, then a week later I statically assign the same one somewhere, and suddenly two devices fight for the IP. Splitting the range up front avoids that entirely.

Here's the layout for `10.10.1.0/24`:

| Range | Purpose | Examples |
|-------|---------|----------|
| `.1` | Gateway / firewall management | OPNsense LAN interface |
| `.2 – .9` | Reserved for future routers / switches | managed switch, APs |
| `.10 – .49` | **Static infrastructure** | Proxmox host, Kubernetes nodes, Pi-hole |
| `.50 – .99` | **Static services** (MetalLB pool) | ingress, Pi-hole DNS VIP, other LB IPs |
| `.100 – .199` | **DHCP pool** (dynamic wired clients) | ethernet-only devices without reservations |
| `.200 – .254` | Reserved | emergencies, break-glass static IPs |

Concrete assignments within the static infra block:

| IP | Host | Role |
|----|------|------|
| `10.10.1.1` | OPNsense (eth3 of Glovary, in VM) | Firewall + gateway + mgmt |
| `10.10.1.10` | `proxmox-glovary` | Hypervisor host |
| `10.10.1.11` | `intel-c-firewall` | Kubernetes control plane (VM on Proxmox) |
| `10.10.1.12` | `amd-w-minipc` | Kubernetes worker (bare metal) |
| `10.10.1.13` | `intel-w-acemagic` | Kubernetes worker (bare metal) |
| `10.10.1.20` | `pihole` | DNS resolver |

The rule I follow: **anything that's referenced by name from another box lives in static space.** Pi-hole is referenced as a DNS server, the Kubernetes nodes are referenced by `kubeadm`/Talos/kubelet, the Proxmox host is referenced by me over SSH. None of them can afford to get a new IP on a reboot. Anything that nobody references back (laptops, phones, my wife's iPad) goes in DHCP and I never think about it again.

In OPNsense I'll configure the DHCP server to only hand out `.100 – .199` and leave the rest alone. No overlap, no surprises.

Confirm the summary, then let it install. Takes two or three minutes on NVMe.

## Step 4: first boot

Once installed, remove the USB and reboot. You'll land on a console showing the web UI URL:

```
https://10.10.1.10:8006
```

Hit that from another machine. Your browser will complain about the self-signed cert — that's expected. Log in as `root` with the password you set, and choose `PAM` as the authentication realm.

![Proxmox web UI login](/images/proxmox-login-placeholder.jpg)

## Step 5: drop the enterprise repos

Fresh installs default to the enterprise repository, which requires a paid subscription. Without it, `apt update` throws a warning and you can't pull updates. Swap in the community repo.

SSH into the node (Proxmox enables SSH by default):

```bash
ssh root@10.10.1.10
```

Disable the enterprise list:

```bash
sed -i 's/^deb/#deb/' /etc/apt/sources.list.d/pve-enterprise.list
sed -i 's/^deb/#deb/' /etc/apt/sources.list.d/ceph.list
```

Add the no-subscription repo:

```bash
echo "deb http://download.proxmox.com/debian/pve bookworm pve-no-subscription" \
  > /etc/apt/sources.list.d/pve-no-subscription.list
```

Then update:

```bash
apt update && apt dist-upgrade -y
```

<Callout type="tip">
The no-subscription repo is free but doesn't get the same QA testing as enterprise. For a homelab it's perfectly fine. For production, pay for a subscription.
</Callout>

## Step 6: suppress the subscription nag

Every time you log into the web UI, Proxmox pops up a "No valid subscription" dialog. It's cosmetic but annoying. The standard fix is a one-liner that patches the JS file:

```bash
sed -i.bak "s/data.status !== 'Active'/false/g" \
  /usr/share/javascript/proxmox-widget-toolkit/proxmoxlib.js
systemctl restart pveproxy.service
```

This gets reverted on every Proxmox update, so I keep the command in a script. Open the web UI in an incognito window to confirm the dialog is gone.

## Step 7: create your first VM

Now for the fun part. In the web UI, click **Create VM** and fill in the wizard. The defaults are sensible for Linux guests. A few things I always change:

- **CPU type:** `host` for maximum performance (tells KVM to expose the physical CPU flags to the guest). Only downside is it breaks live migration to dissimilar hardware, which doesn't matter on a single-node cluster.
- **Machine type:** `q35` for anything modern. `i440fx` is fine for very old guests.
- **Disk cache:** leave at default (`none`) unless you have a specific reason.
- **Network model:** `VirtIO` — the paravirtualized driver is dramatically faster than emulated hardware.

For the Kubernetes control plane VM I gave it 4 vCPUs, 4 GB RAM, and 32 GB of disk. Talos Linux is lightweight and doesn't need more.

## Gotchas I hit

**BIOS virtualization was off by default.** The Glovary shipped with VT-x disabled. The Proxmox installer didn't complain, but VMs refused to start until I flipped the BIOS flag.

**The default bridge assumes DHCP.** If your network uses static IPs (mine does for infrastructure nodes), configure `vmbr0` explicitly during install rather than fighting it later.

**`/etc/hosts` needs the hostname.** If the FQDN doesn't resolve to the node's IP, the web UI starts throwing weird errors after a reboot. Double-check this before you walk away.

**Cluster joins are one-way.** If you plan to run multiple Proxmox nodes together, set up the cluster before putting VMs on the secondary nodes — joining a cluster wipes existing VM config on the joining node.

## What's next

With Proxmox running, the Glovary now hosts:

- **OPNsense VM:** 2 vCPU, 2 GB RAM, passthrough access to the four 2.5GbE NICs
- **Kubernetes control plane VM:** 4 vCPU, 4 GB RAM, running Talos Linux

The beauty of this setup is that OPNsense and the control plane are completely isolated. A problem with one doesn't take down the other, and I can snapshot either VM before a risky change.
